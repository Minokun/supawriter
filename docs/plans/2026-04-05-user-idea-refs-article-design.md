# Design: 用户想法 + 参考资料 → 专业文章

Generated by /office-hours on 2026-04-05

Status: APPROVED (post review)
Mode: Startup

## Problem Statement

SupaWriter 当前文章生成流程是 `topic → 搜索 → 大纲 → 写作`。用户只能输入一个标题，系统自动搜索后写文章。内容创作者真正需要的是：我有自己的观点和参考资料，需要 AI 帮我搜索补充更多素材，然后基于**我的观点**写一篇专业文章。

核心痛点：整理资料太耗时。现有工具（Notion AI、ChatGPT 等）不够好。

## Target User

内容创作者（自媒体博主、微信公众号作者、知乎创作者等）。

## Narrowest Wedge

用户输入想法/观点 + 贴入参考资料文字，系统搜索补充内容后写出专业文章。MVP 不做文件上传，只做文字粘贴。

## Constraints

- 复用现有 `ArticleGeneratorService.generate_article_stream()` 管线，不新建独立管线
- 不改数据库 schema
- 不改 worker/agent 路径
- 两个新字段都是可选的，不填走现有流程

## Detailed Design (post review)

### 改动清单

#### 1. `backend/api/routes/articles_enhanced.py`

`ArticleGenerateRequest` 模型新增：
```python
user_idea: Optional[str] = Field(
    None, max_length=5000,
    description="用户的想法/观点（用于指导文章方向，最多5000字）"
)
user_references: Optional[str] = Field(
    None, max_length=50000,
    description="用户贴入的参考文字（系统将搜索补充素材后结合生成文章，最多50000字）"
)
```

Pydantic validator 规范化空字符串：
```python
@field_validator('user_idea', 'user_references')
@classmethod
def normalize_empty(cls, v):
    if v is not None and v.strip() == '':
        return None
    return v
```

**两个调用点都要传参**：
- `_run_generation()` (~line 181): POST /generate 的后台任务
- `event_generator()` (~line 403): POST /generate/stream 的 SSE 路径

#### 2. `backend/api/services/article_generator.py`

`generate_article_stream()` 方法：
- 新增参数: `user_references: Optional[str] = None`, `user_idea: Optional[str] = None`
- 搜索后、大纲生成前，如果 `user_references` 有内容：
  - 合成 SearchResult 字典，设置完整字段：
    ```python
    {
        'title': '用户参考资料',
        'html_content': user_references,
        'url': '',
        'images': [],
        'source': 'user',
        'relevance_score': 999  # 确保不被丢弃
    }
    ```
  - 插入到 `search_results` 列表头部
- `user_idea` 通过 XML 标签注入到 prompt 的 question 参数中：
  - 大纲生成时: question 前加 `<user_idea>{user_idea}</user_idea>`
  - 章节写作时: 同上
- `references_list` 构建时过滤 `source == 'user'` 的条目
- `search_result_items`（进度推送）中也要过滤

#### 3. `utils/prompt_template.py`

**不改 prompt 模板字符串**。`user_idea` 通过 XML 标签注入到 question 参数中，不是模板占位符。

#### 4. `utils/searxng_utils.py`

`Search.get_search_result()` 新增参数 `user_idea: Optional[str] = None`：
- 作为 keyword argument 添加，向后兼容
- 在 LLM query 优化时，将 user_idea 融合到优化 prompt 中（如作为 system prompt 的一部分）

#### 5. `frontend/src/components/writer/WriterForm.tsx`

在 topic 输入框和 customStyle 之间插入两个 textarea：

- `user_idea`: rows=2, maxLength=2000, placeholder="输入你对这个主题的观点或角度"
- `user_references`: rows=3, maxLength=10000, placeholder="粘贴参考资料文字，系统会结合你的资料和搜索内容生成文章"

两个都 `(可选)`，始终可见，不做折叠。
提交时 `.trim()` 后为空则发 `undefined`。
成功提交后 reset 状态。

#### 6. 前端类型定义（两处都要改）

**`frontend/src/types/api.ts`** `ArticleGenerateRequest`:

```typescript
user_idea?: string;
user_references?: string;
```

**`frontend/src/lib/api/articles.ts`** — 如果有独立的 `ArticleGenerateRequest` 定义也要同步更新，或改为从 `@/types/api.ts` 统一导入。

### 数据流

```
用户输入:
  topic: "AI 对教育的影响"
  user_idea: "我认为 AI 不会取代老师，但会改变老师的教学方式"
  user_references: "（用户贴入的一段关于某学校试点 AI 教学的报道文字）"

系统流程:
  1. user_references → 合成 SearchResult（source='user', relevance_score=999）→ 插入列表头
  2. topic + user_idea → 搜索 query 优化 → 搜索补充内容
  3. 用户资料 + 搜索结果 → llm_task() 并行处理（用户资料权重最高）
  4. 大纲生成：question 中注入 <user_idea>...</user_idea> 标签
  5. 逐章写作：每章 question 中注入 <user_idea>...</user_idea>
  6. references_list 过滤掉 source='user' 的条目
  7. 输出专业文章
```

### 测试策略

1. **单元测试**: `ArticleGenerateRequest` 模型验证（max_length、空字符串规范化）
2. **单元测试**: 合成 SearchResult 构造（完整字段、relevance_score）
3. **集成测试**: `generate_article_stream()` 带 mock Search 和 mock llm_task
4. **边界测试**: 超长引用、特殊字符、XML 标签冲突
5. **回归测试**: 不填新字段时输出与之前完全一致

## Success Criteria

1. 不填 user_idea 和 user_references → 和现有流程完全一样
2. 只填 user_idea → 搜索结果融合用户观点生成大纲和文章
3. 只填 user_references → 参考资料 + 搜索补充 → 文章
4. 两个都填 → 完整新流程
5. 前端 WriterForm 看到新字段
6. SSE 流正常返回进度
7. references_list 不包含空 URL 的用户资料条目
